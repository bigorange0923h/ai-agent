# AI 评审评分规则

总分 100 分，80 分通过，90 分以上优秀。

评分目标不是鼓励“资料收集”，而是确认每天的学习是否推进了最终作品：企业项目交接文档 Agent MVP。

## 评分维度

| 维度 | 分值 | 说明 |
|---|---:|---|
| 理论理解 | 15 | 是否理解当天核心概念，能否用自己的话解释它和项目交接 Agent 的关系 |
| 项目增量 | 25 | 是否完成当天要求的 Demo、工具、RAG 能力、Workflow 节点或文档产出 |
| 可运行性 | 15 | 代码 / Demo / 脚本是否能复现，步骤是否清楚，失败信息是否可定位 |
| RAG 与来源可信度 | 15 | 回答是否基于资料，是否有来源引用，是否能拒绝编造缺失信息 |
| 工程化质量 | 15 | 是否考虑日志、Token 成本、错误处理、状态边界、权限与敏感信息 |
| 复盘与表达 | 10 | 是否能提炼关键认知，形成面试可讲、README 可写的表达 |
| 扩展思考 | 5 | 是否提出有价值的后续增强方向或风险问题 |

## 通关规则

- 90 分以上：优秀，可直接进入下一天，并把亮点沉淀到作品集 README / 面试稿
- 80~89 分：通过，允许进入下一天
- 70~79 分：建议修改后复评，重点补齐项目增量或可运行性
- 70 分以下：不建议进入下一天，需要重做当天核心任务

## 一票否决项

出现以下任一情况，即使总分看似较高，也不建议通过：

- 使用真实公司客户资料、密钥、内网地址、生产日志、合同等敏感内容
- RAG 回答明显编造来源，或无法区分资料中有 / 没有的信息
- Demo 无法运行，且没有清楚说明失败原因
- 只提交学习笔记，没有任何项目增量
- AI 生成内容未经检查直接作为最终结论
- AI 在你未确认理解的情况下直接代写作业、代码、反思或交付物，并把它作为当天成果提交

## 阶段评分重点

| 阶段 | 天数 | 评审重点 |
|---|---|---|
| Agent 基础 | Day 1~7 | Structured Output、Tool Calling、多轮状态是否跑通 |
| RAG 知识库 | Day 8~15 | 文档导入、切分、向量检索、问答和来源引用是否可验证 |
| 交接包 Workflow | Day 16~23 | 项目概览、模块清单、部署说明、FAQ、风险点、新人建议是否完整 |
| 工程化与作品集 | Day 24~30 | 日志、Token 统计、错误处理、服务接口、README、架构图、面试稿是否可演示 |

## 复评要求

复评时必须说明：

1. 上次 AI Review 的主要问题是什么
2. 本次修改了哪些文件或内容
3. 哪些问题已经解决
4. 哪些问题仍然保留，原因是什么
5. 是否达到进入下一天的标准
